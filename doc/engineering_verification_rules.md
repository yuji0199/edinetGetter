# エンジニアリング動作確認規定 (Engineering Verification Rules)

<div class="log">
2026-03-26: 初版策定。完了定義に「動作確認」を必須項目として追加。
</div>

## 1. 目的
本規定は、コードの変更が既存のシステムを破壊せず、かつ新規機能が期待通りに動作することを保証することを目的とする。いかなるタスクも、本規定に基づく検証が完了するまでは「完了」と見なさない。

## 2. サーバー起動確認 (Server Startup Protocol)
変更を適用した後は、必ず以下の手順でサーバーの状態を確認せよ。

- **バックエンド (FastAPI)**:
    - `venv/bin/uvicorn main:app --reload --port 8000` を実行し、`Application startup complete.` がログに表示されることを確認する。
    - インポートエラー（`ImportError`, `ModuleNotFoundError`）が発生していないか、スタックトレースを最下部まで確認する。
- **フロントエンド (Vite)**:
    - `npm run dev` を実行し、ブラウザでアクセス可能なURLが表示されることを確認する。
    - コンソールにビルドエラーが表示されていないか確認する。

## 3. インポート規約 (Import Convention)
バックエンド（FastAPI）において、相対インポートによるエラーを避けるため、以下のルールを遵守せよ。

- **絶対インポートの推奨**:
    - `from ..database import get_db` のような相対インポートは避け、`import database, models, schemas` または `from services.growth import ...` のように、`backend` ディレクトリを基点としたインポートを行う。
- **実行環境の同一性**:
    - 常に `backend` ディレクトリをカレントディレクトリ（CWD）としてサーバーを起動する。

## 4. 機能検証 (Functional Validation)
機能の完了報告前に、以下のいずれかの方法で動作を証明せよ。

- **ブラウザサブエージェントによる検証**:
    - 実際に画面を操作し、期待するデータが表示されるか、エラーが発生しないかをスクリーンショットまたは録画で記録する。
- **APIスモークテスト**:
    - `curl` またはテストスクリプトを使用し、変更したエンドポイントが `200 OK` を返し、正しいレスポンスボディを返すことを確認する。

## 5. 完了定義 (Definition of Done)
以下の項目がすべてチェックされた状態を「完了」とする。

- [ ] `task.md` の全項目が完了している。
- [ ] サーバーがエラーなく起動する（バックエンド・フロントエンド）。
- [ ] 変更箇所について、少なくとも1回以上の正常系テストが実行されている。
- [ ] 画面キャプチャまたはログによる「動く証拠」が `walkthrough.md` に記載されている。
