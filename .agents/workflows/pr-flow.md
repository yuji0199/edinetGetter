これは、新しいIssue（要件）に基づいて機能開発や修正を行う際の、私たち（AIとユーザー）の標準ワークフローです。
チャットから「/pr-flow」や「PRワークフローで始めて」と入力された際にこのワークフローを発動し、すべてのステップを順番に守って進行してください。

## Step 1: ブランチ作成と事前調査（Plan/Research）
1. ユーザーから解決すべきIssueや要件が提示されます。
// turbo
2. 現在の `main` ブランチの最新状態を取り込みます (`git checkout main && git pull`).
3. 要件に基づいた作業用ブランチを作成します（例: `git checkout -b feature/issue-XX` または `fix/〇〇`）。
4. （必要な場合）関連するファイルを調査し、システムへの影響を調べます。まだコードは書き換えません。

## Step 2: 実装計画の作成と合意 (Approval)
1. `implementation_plan.md` を作成し、システムのどの部分（テーブル、API、UIなど）を変更するか、具体的な設計を記述します。
2. ユーザーからの承認（Approve）または修正指示（Change Request）を待ちます。

## Step 3: 実装の実行 (Execution)
1. ユーザーから承認を得たら、`task.md` を用いてタスクを分解し、コードの書き換え（実装）を開始します。
2. 変更が完了したら、ローカルでの動作確認（ターミナルへのスクリプト実行等）を行います。
3. エラー等がないことを確認したら、コミットを作成します（機能ごとに小さく綺麗なコミットメッセージを心がけます）。

## Step 4: プッシュとPRの作成 (Pull Request)
1. 作業完了をユーザーに報告する前に、変更を作業ブランチ（origin）にプッシュします。
2. GitHub上（あるいは `gh` コマンド等）で対象ブランチから `main` ブランチへ向けたPull Request（PR）を作成するようユーザーに促すか、権限があればエージェントである私が自動作成します。
3. ユーザーに対し、GitHubのPRリンクにて「コード差分（Diff）」を確認・レビューするよう依頼し、完了レポートとして `walkthrough.md` を提示します。

## Step 5: レビュー対応とマージ
1. ユーザーからPRへの修正指示があれば、Step 3 に戻り修正・Pushを行います（PRは自動でアップデートされます）。
2. Approveされマージされた旨が通知されたら、次のタスクに向けて待機します。
